Wildcards

The wildcard (“*”) matches any element tag. You cannot use the wildcard to match attributes.

Sometimes you need to match an element with a unique quality and the kind of element is unimportant. For example, your style guide recommends using a true em-dash character (“—”) instead of a double hyphen (“--”). However, it is acceptable to use a double-hyphen in programming code, which is in pre elements. Therefore, the when sections have to enumerate each of the elements for which a double-hyphen could incorrectly appear:

when: self-text p "--"
when: self-text li "--"
when: self text td "--"
when: self-text h1 "--"
when: self-text h2 "--"
when: self-text h3 "--"
when: self-text h4 "--"
when: self-text h5 "--"
when: self-text h6 "--"

test: fail

This is a tedious, error-prone way to match double-hyphens, especially because the type of element is not important but the element’s content is. You can simplify these sections to a single when section with a wildcard (“*”) and enumerate the only case where it’s acceptable in the test section:

when: self-text * "--"

test: self pre

This simplified when section matches any element that contains a double-hyphen then the test section ensures the double-hyphen appears in the only acceptable element, pre.

Sometimes you can simplify a rule with a wildcard in a test section. For example, if your team does not allow exclamation marks at the end of paragraphs, list items, and table cells, you would write the following when sections:

when: self p
when: self li
when: self td

To test these elements, you need to write the following corresponding test sections:

test: not self-ends p "!"
test: not self-ends li "!"
test: not self-ends td "!"

You can reduce the test sections to a single test section by using the wildcard (“*”):

test: not self-ends * "!"

The simplified test section matches any of the elements found in the when sections.
